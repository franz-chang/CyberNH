# 从激活向量到安全特征候选：一次基于 Qwen3-VL SAE 的探究过程

## 引言

在解释大模型时，一个很自然的问题是：当模型面对安全和不安全请求时，它内部的激活向量到底发生了什么变化？这些变化只是整体幅度的波动，还是已经在某些层、某些稀疏特征上形成了相对稳定的分离结构？

围绕这个问题，我们没有一开始就直接训练分类器，也没有急着做干预实验，而是先把任务拆成一个更可控的分析流程：先看不同 prompt 组在模型内部是否真的能分开，再决定下一步应该分析哪些层、哪些特征、以及用什么指标来筛选候选 feature。

这篇文章梳理的，就是这条探究路径。

## 一、问题定义：我们到底想找什么

最初的问题并不是“如何构建一个安全分类器”，而是更基础的一层：

- 安全请求与不安全请求，是否会在模型内部形成可观察的表征差异？
- 这种差异更适合在原始 hidden state 空间中看，还是在 SAE 稀疏特征空间中看？
- 如果要继续往下做 token-level inspection 或 steering，优先应该从哪些层开始？

这决定了我们的第一步不能直接追求任务性能，而应该先做表征分析。

## 二、为什么采用 SAE 视角

原始激活向量虽然信息量很大，但维度高、可解释性弱。相比之下，SAE 能把 dense hidden states 编码成稀疏 feature activations，使得后续分析更容易落到“哪些 feature 更常被触发、哪些 feature 在某类 prompt 上更强”这种层面。

因此，我们采用了两层表示：

1. 原始 hidden states
   用来判断两组输入在该层的整体方向和幅度是否不同。

2. SAE feature activations
   用来判断两组输入在稀疏特征空间中是否发生了更清晰的分离。

这样的设计有一个重要好处：如果某层在原始激活空间和 SAE 特征空间里都表现出差异，我们就更有理由把它视为后续分析候选层；反之，如果只有幅度很大但方向几乎没变，就要谨慎解释。

## 三、第一步：先做 prompt 对照实验

为了避免一开始就陷入大规模数据处理，我们先采用一个小而清晰的对照实验。对照组来自两类 prompt：

- `safe`
- `unsafe`

这一步的目标不是追求统计显著性，而是用一组构造明确的样本先观察趋势。我们关心的问题是：

- 模型在处理这两组文本时，某一层的 hidden states 是否明显不同？
- 把这些 hidden states 编码进 SAE 之后，feature 空间里的差异是否比原始空间更清楚？

为此，我们在 Qwen3-VL 的文本层上逐层抓取激活，并把每个 prompt 最后若干个 token 位置作为分析对象。这么做是因为句尾通常更接近模型对整条指令形成内部表征后的状态，比较适合做第一轮安全对照观察。

## 四、第二步：做 layer sweep，而不是只盯住单层

如果一开始只挑一层来分析，风险很大。因为我们并不知道：

- 差异究竟出现在早层还是中后层
- 不同层的差异到底是方向分离，还是仅仅是尺度变化
- 哪些层的 SAE 特征最值得后续解释

所以第二步的关键决策，是先做一轮 layer sweep。也就是对多个层重复同样的 safe vs unsafe 对照，并比较它们的分离程度。

在这一阶段，我们主要用了几类指标：

- 原始激活 centroid cosine
- 原始激活 centroid L2 distance
- SAE 特征 centroid cosine
- SAE 特征 centroid L2 distance
- top-feature Jaccard

这些指标分别回答不同问题：

- `cosine` 更适合看方向是否不同
- `L2` 更容易反映幅度变化
- `top-feature Jaccard` 则可以看两组输入激活最高的特征集合是否发生明显更替

## 五、第三步：区分“真正的分离”和“只是变大了”

layer sweep 之后，一个非常重要的发现出现了：并不是所有“差异大”的层都值得深入分析。

在一些中后层里，我们确实看到 `L2` 距离变大，但同时 `cosine` 又非常接近 1。换句话说，这些层上两组激活的方向几乎没变，只是整体幅度拉开了。这类结果如果直接解释成“安全语义在这里最强”，其实是有风险的。

相反，真正值得优先关注的层，是那些满足以下条件的层：

- SAE cosine 明显下降
- top-feature Jaccard 有分化
- 差异不是单纯由幅度拉高造成的

这一步帮助我们从“所有层都可能有信息”的宽泛状态，收敛到“哪些层最有解释价值”的更具体结论。

## 六、第四步：把候选层收敛到 `layer0`、`layer4`、`layer5`

根据现有实验结果，`safety` 对照最值得继续往下看的层，主要集中在：

- `layer0`
- `layer4`
- `layer5`

原因并不复杂，但很关键：

- `layer0` 的 SAE cosine 最低，说明 safe/unsafe 在这个层上的方向分离最明显
- `layer4` 和 `layer5` 仍然维持了较好的可解释分离
- 再往后的很多层虽然数值差异大，但更像“幅度变化”，不适合作为第一批安全 feature 的主分析层

也就是说，到这一步为止，我们已经完成了一个很重要的收缩：从“整个模型的很多层”收缩到“三个最值得看的层”。

## 七、第五步：从层分析转向 feature ranking

确定了候选层之后，问题自然会变成：这些层里，到底是哪些 SAE feature 在 unsafe prompt 上更活跃？

这时再继续看层级别的 centroid 指标就不够了，因为它们只能告诉我们“两组整体不同”，却不能告诉我们“究竟是哪些特征在起作用”。

因此，下一步分析必须转向 feature ranking。

这里我们采用的核心指标是：

```text
unsafe_mean - safe_mean
```

具体做法是：

1. 在某一层收集所有 `safe` prompt 的 SAE feature activations
2. 在同一层收集所有 `unsafe` prompt 的 SAE feature activations
3. 对每个 feature 分别计算两组的平均激活值
4. 用 `unsafe_mean - safe_mean` 作为该 feature 的差异分数
5. 按这个分数排序，找出最偏向 `unsafe` 的 feature

这个思路的优点是非常直接。它不要求我们先有复杂模型，也不要求先定义“危险 feature”的强标签，而是先从统计上找出“哪些 feature 在不安全样本中更高”。

## 八、为什么 `unsafe_mean - safe_mean` 是一个合适的起点

这个指标之所以适合作为第一版分析工具，主要有三个原因。

第一，它可解释。
对每个 feature 来说，分数越大，意味着它在 `unsafe` 组上平均更高；分数越小，则说明它更偏向 `safe` 组。这个解释非常直观。

第二，它和 SAE 的稀疏性很匹配。
因为 SAE feature 本身就是稀疏激活，所以对 feature 维度做组间均值比较，能够很自然地回答“这类输入更容易激活哪些特征”。

第三，它适合做候选发现。
在当前这个阶段，我们并不是要发布一个最终结论，而是要为后续的 token-level inspection 和 steering 找到第一批候选 feature。这个指标足够轻量，也足够高效。

## 九、为什么还要加入非零率、标准差和效应量

不过，只看均值差还不够稳。

一个 feature 的 `unsafe_mean - safe_mean` 很大，可能有很多原因：

- 它在大多数 unsafe token 上都被稳定激活
- 它只在少数 token 上激活，但幅度特别高
- 它本身波动很大，均值差只是偶然被少数样本拉出来

所以，在 feature ranking 阶段，我们计划同时计算几类辅助指标：

- 非零激活率
- 标准差
- effect size

这样做的目的，是把“均值更高”和“稳定更高”区分开。

如果某个 feature 同时满足：

- `delta` 大
- `unsafe_nonzero_rate` 明显更高
- `effect_size` 也高

那么它就更像一个稳定的 `unsafe` 候选 feature，而不是偶然波动。

## 十、第六步：把 ranking 结果变成下一轮问题

到这里，分析工作还没有结束，反而只是进入更细的一层。

因为 feature ranking 只会告诉我们：

- 哪些 feature 值得看

但它还不能回答：

- 这些 feature 究竟对应哪些 token
- 它们是响应“危险动作词”，还是响应“命令语气”
- 它们是否只是被某类格式、符号或局部模式误触发

所以，feature ranking 的真正价值，不是给出最后答案，而是把问题继续往下推进到 token-level inspection。

换句话说，当前这条探究路径的结构其实是：

1. 用 prompt 对照确认“有差异”
2. 用 layer sweep 确认“差异主要在哪些层”
3. 用 feature ranking 确认“差异主要落在哪些特征”
4. 再用 token-level inspection 确认“这些特征到底在响应什么”

## 十一、这条路径为什么重要

这条路径看起来比较慢，但它的好处在于每一步都尽量减少误判。

如果没有 layer sweep，我们可能会盲目地在后层追逐大 `L2` 值。

如果没有 SAE feature ranking，我们可能只能停留在“这两组激活不同”这种过于粗糙的结论上。

如果没有后续的 token-level inspection，我们又可能把某些统计上偏高的 feature 误认为“真正的安全语义 feature”。

因此，当前的工作重点其实不是“尽快做出一个强结论”，而是建立一条从激活向量出发、逐步走向可解释 feature 的可靠路径。

## 十二、当前阶段的结论

截至目前，这个问题的探究已经收敛出几条较明确的结论。

第一，safe 与 unsafe prompt 的差异是可以在模型内部观察到的，但不同层上的性质不一样。
有些层更像方向分离，有些层更像幅度变化，这两者不能混为一谈。

第二，`layer0`、`layer4`、`layer5` 是当前最值得继续做安全 feature 分析的层。
它们在已有 sweep 中表现出更有解释价值的分离。

第三，下一步最自然的工作不是直接做干预，而是先在这三层上做 `unsafe_mean - safe_mean` feature ranking。
这样可以从层级别分析进一步推进到 feature 级别分析。

## 结语

从激活向量出发去理解安全行为，最难的地方往往不是“算不出指标”，而是“如何避免过早下结论”。当前这套分析流程的核心价值，就在于它把一个原本模糊的问题，拆成了一系列更小、更稳、更容易验证的步骤。

我们并没有直接问模型“哪里有安全特征”，而是先问：

- 差异是否存在？
- 差异出现在哪些层？
- 这些层上的差异是方向性的还是幅度性的？
- 哪些 SAE feature 真正值得继续解释？

只有沿着这条路径逐步推进，后续的 token-level inspection、probe 乃至 steering，才会建立在更可靠的基础上。
